## 1. Foundational GenAI Concepts

### Tokens
- The **basic unit of text** that a language model reads and processes.
- A token is roughly **4 characters** or **¾ of a word** in English.
- Text is broken into tokens **before** being fed into a model (tokenization).
- **Examples**:
  - "Hello" → 1 token
  - "Unbelievable" → 3–4 tokens
  - "AWS is great!" → ~5 tokens
- **Why it matters**:
  - LLMs have a **context window** — a maximum number of tokens they can process at once (input + output combined).
  - API pricing is typically based on **tokens consumed** (input tokens + output tokens).
  - Longer prompts = more tokens = higher cost and latency.

> 💡 **Exam Tip**: Tokens are the unit of both **processing and billing** for LLMs. A larger context window = the model can handle longer documents and multi-turn conversations.

---

### Chunking
- The process of **splitting large documents or text** into smaller, manageable pieces (chunks) before processing or storing them.
- Essential when documents exceed a model's context window.
- Chunks are typically converted to embeddings and stored in a **vector database** for retrieval.

**Chunking strategies:**

| Strategy | Description | Best For |
|---|---|---|
| **Fixed-size** | Split every N tokens/characters | Simple, predictable pipelines |
| **Sentence/paragraph-based** | Split at natural sentence or paragraph boundaries | Preserving grammatical context |
| **Semantic chunking** | Split at meaningful topic boundaries | High-quality retrieval accuracy |
| **Sliding window** | Overlapping chunks to preserve context at boundaries | Avoiding lost context at split points |

- **Why it matters**: Critical in **RAG pipelines** — documents must be chunked before being indexed and retrieved.

> 💡 **Exam Tip**: Chunking is a key pre-processing step in **RAG architectures**. The chunk size and strategy directly impacts retrieval quality.

---

### Embeddings
- **Numerical vector representations** of text (or other data) that capture **semantic meaning**.
- Similar concepts produce embeddings that are **geometrically close** in vector space.
- Generated by **embedding models** (e.g., Amazon Titan Embeddings, OpenAI Ada).

**Example of semantic closeness:**
```
Embedding distance:
"king" ←→ "queen"   → CLOSE (related concepts)
"king" ←→ "banana"  → FAR   (unrelated concepts)
```

**Why embeddings matter:**
- Power **semantic search** — find results by meaning, not just keyword match.
- Core component of RAG: chunks → embeddings → stored in vector database.
- Enable similarity comparisons, clustering, and recommendations.
- Allow models to work with **non-text modalities** (images, audio) by mapping them into the same vector space.

> 💡 **Exam Tip**: Embeddings = **meaning captured as numbers**. They enable semantic similarity search — the foundation of modern AI-powered retrieval.

---

### Vectors
- A vector is simply a **list of numbers** (e.g., `[0.21, -0.85, 0.44, ...]`) used to represent data mathematically.
- In GenAI, embeddings ARE vectors — the terms are often used interchangeably.
- **Vector similarity** is measured using:
  - **Cosine similarity**: Angle between vectors (most common for text)
  - **Euclidean distance**: Straight-line distance between points
  - **Dot product**: Measures alignment of vectors
- Vectors are stored in **vector databases** (e.g., Amazon OpenSearch, Aurora with pgvector) for fast similarity search.

> 💡 **Exam Tip**: Vectors are the mathematical container for embeddings. When you search a vector database, you're finding vectors (and their associated content) that are numerically similar to your query vector.

---

### Prompt Engineering
- The practice of **designing and refining the input (prompt)** to a model to guide it toward producing accurate, relevant, and well-formatted outputs.
- Does **not** require changing model weights — it works entirely at inference time.
- Key techniques: zero-shot, few-shot, chain-of-thought, instruction formatting (covered in detail in Task 3.2).
- **Why it matters**: Small changes in prompt wording can significantly change model output quality.

---

### Transformer-Based LLMs
- The **dominant architecture** behind modern language models (BERT, GPT, Claude, Llama, etc.).
- Introduced in the 2017 paper *"Attention Is All You Need"* by Google researchers.

**Key transformer components:**

| Component | Role |
|---|---|
| **Self-Attention Mechanism** | Allows the model to weigh the importance of every word relative to every other word in the input |
| **Positional Encoding** | Gives the model a sense of word order (since transformers process all tokens in parallel) |
| **Encoder** | Reads and understands input text (used in models like BERT) |
| **Decoder** | Generates output text token by token (used in GPT-style models) |
| **Encoder-Decoder** | Both reads and generates — used for translation, summarization (e.g., T5) |

**Why transformers revolutionized AI:**
- Process entire sequences **in parallel** (unlike older RNNs that processed word-by-word).
- Scale efficiently — more data + more compute = dramatically better performance.
- The self-attention mechanism allows the model to understand **long-range dependencies** in text.

**LLM types:**
- **Autoregressive (Decoder-only)**: Generates text left-to-right, one token at a time. Most modern chat models (GPT, Claude, Llama). Best for generation tasks.
- **Masked (Encoder-only)**: Trained to predict masked tokens in a sentence. Best for understanding/classification (BERT).
- **Sequence-to-Sequence (Encoder-Decoder)**: Reads full input, generates full output. Best for translation, summarization (T5, BART).

> 💡 **Exam Tip**: Most modern LLMs used for chat and generation (Claude, GPT, Llama) are **decoder-only transformer** models. The self-attention mechanism is what makes transformers powerful — it lets every token "attend" to every other token.

---

### Foundation Models (FMs)
- Large, **general-purpose models trained on massive datasets** that can be adapted for a wide range of downstream tasks.
- Trained once at enormous scale → deployed and reused across many applications (via prompting, fine-tuning, or RAG).
- **Characteristics of FMs**:
  - Trained on broad, diverse data (text, code, images, etc.)
  - Emergent capabilities: abilities that appear at scale not seen in smaller models
  - Adaptable: can be fine-tuned or prompted for specific tasks
  - Accessible via APIs (no need to manage infrastructure)
- **Examples**: Claude (Anthropic), GPT-4 (OpenAI), Llama (Meta), Amazon Titan, Stable Diffusion.
- **AWS access point**: Amazon Bedrock provides access to multiple FMs via a single API.

---

### Multimodal Models
- Models that can **process and/or generate multiple types of data** (modalities) — text, images, audio, video, code.
- **Input multimodal**: Accept multiple types of input (e.g., text + image → text answer).
- **Output multimodal**: Generate multiple types of output (e.g., text → image).

**Examples by capability:**

| Capability | Example Models | Use Case |
|---|---|---|
| Text + Image → Text | Claude 3, GPT-4V | Visual Q&A, image captioning |
| Text → Image | Stable Diffusion, DALL-E, Titan Image | Image generation |
| Text + Audio → Text | Whisper | Speech recognition with context |
| Text → Audio | Amazon Polly | Text-to-speech |
| Text + Video → Text | Gemini 1.5 | Video understanding |

> 💡 **Exam Tip**: Multimodal = handles **more than one type of data**. Know that Claude 3 and GPT-4V can accept both text and images as input. Image *generation* models (Stable Diffusion, DALL-E) are also multimodal.

---

### Diffusion Models
- A class of **generative models** used primarily for **image, audio, and video generation**.
- Work by learning to **reverse a noising process**:
  1. **Forward process**: Gradually add random noise to training images until they become pure noise.
  2. **Reverse process (learned)**: Train the model to progressively **denoise** — remove noise step by step to reconstruct an image.
  3. At inference: Start from random noise, apply learned denoising steps, arrive at a generated image.

```
Training:   [Real Image] → [+ noise] → [+ noise] → [Pure Noise]
Inference:  [Pure Noise] → [- noise] → [- noise] → [Generated Image]
```

- **Text-to-image**: Condition the denoising process on a text prompt to generate matching images.
- **Examples**: Stable Diffusion, DALL-E, Midjourney, Amazon Titan Image Generator.
- **Contrast with LLMs**: LLMs generate token-by-token; diffusion models generate by iterative denoising.

> 💡 **Exam Tip**: Diffusion models = **iterative denoising** for image/audio/video generation. They are the foundation of text-to-image generation tools.

## 2. GenAI Use Cases

### Image Generation
- Generate realistic or artistic images from text descriptions.
- **Examples**: Product visualization, marketing assets, concept art, synthetic training data.
- **AWS**: Amazon Titan Image Generator (via Amazon Bedrock), Stability AI models on Bedrock.

### Video Generation
- Generate or edit video content from text prompts or existing footage.
- **Examples**: Short-form content creation, video game asset generation, synthetic training videos.
- Emerging capability — models like Sora (OpenAI) and Runway.

### Audio Generation
- Generate speech, music, or sound effects from text or other inputs.
- **Examples**: Voiceovers, podcast production, background music, audiobooks.
- **AWS**: Amazon Polly (TTS), third-party audio models on Bedrock.

### Summarization
- Condense long documents, articles, or conversations into concise summaries.
- **Examples**: Meeting notes, legal document summaries, news digests, customer feedback analysis.
- **AWS**: Any text-generation FM on Amazon Bedrock.

### AI Assistants / Chatbots
- Conversational agents that answer questions, provide guidance, and assist with tasks.
- **Examples**: Customer service bots, internal knowledge assistants, personal productivity tools.
- **AWS**: Amazon Lex + Bedrock FMs, Amazon Q Business.

### Translation
- Convert text from one language to another, often with contextual nuance.
- **Examples**: Localizing software, translating support content, multilingual chatbots.
- **AWS**: Amazon Translate (managed NMT service), or text FMs for nuanced translation.

### Code Generation
- Generate, complete, explain, debug, or review code from natural language descriptions.
- **Examples**: Autocomplete in IDEs, converting pseudocode to working code, generating unit tests.
- **AWS**: Amazon CodeWhisperer (now Amazon Q Developer), code-capable FMs on Bedrock.

### Customer Service Agents
- AI-powered agents that handle customer queries, route tickets, and resolve issues autonomously.
- **Examples**: Returns processing, order status, technical troubleshooting bots.
- **AWS**: Amazon Bedrock Agents + Knowledge Bases + Lex.

### Search
- Semantic search that retrieves results based on **meaning** rather than keyword matching.
- **Examples**: Enterprise document search, product search, knowledge base Q&A.
- **AWS**: Amazon Kendra (intelligent search), RAG with Amazon Bedrock + OpenSearch.

### Recommendation Engines
- Suggest relevant items, content, or actions based on user behavior and preferences.
- **Examples**: Product recommendations, content personalization, next-best-action in sales.
- **AWS**: Amazon Personalize, or LLM-powered recommendation logic on Bedrock.

## 3. The Foundation Model Lifecycle

The FM lifecycle describes the **end-to-end process** from raw data to a deployed, continuously improving model.

```
┌──────────────────────────────────────────────────────────────────┐
│                   Foundation Model Lifecycle                      │
│                                                                    │
│  [1] Data        [2] Model      [3] Pre-        [4] Fine-         │
│  Selection   →   Selection  →   Training    →   Tuning           │
│                                                                    │
│  [5] Evaluation → [6] Deployment → [7] Feedback & Monitoring      │
│         ↑_______________________________________↓                  │
│                   (Continuous Improvement Loop)                    │
└──────────────────────────────────────────────────────────────────┘
```

---

### Stage 1: Data Selection
- Identify and collect the raw data the model will be trained on.
- **For pre-training**: Massive, diverse datasets — web crawls (Common Crawl), books, code, scientific papers, Wikipedia.
- **For fine-tuning**: Smaller, curated, task-specific labeled datasets.
- **Key considerations**:
  - Quality: Clean, accurate, well-formatted data.
  - Diversity: Broad representation to reduce bias.
  - Licensing: Data must be legally usable for training.
  - Volume: Enough data to learn meaningful patterns.
  - Privacy: PII and sensitive data must be removed or anonymized.

---

### Stage 2: Model Selection
- Choose the **right model architecture and size** for the task before training or fine-tuning.
- **Considerations**:
  - Task type: Text generation, image generation, multimodal?
  - Compute budget: Larger models need more GPU/TPU resources.
  - Latency requirements: Smaller models are faster at inference.
  - Context window needs: Longer documents require larger context windows.
  - Existing FMs: Should you train from scratch or start from a pre-trained FM?
- **AWS context**: On Amazon Bedrock, model selection involves choosing from available FMs (Claude, Titan, Llama, etc.) based on these criteria.

---

### Stage 3: Pre-Training
- Train the model on the large dataset using **self-supervised learning** (e.g., next-token prediction).
- The model learns **general language understanding, knowledge, and reasoning**.
- Extremely compute-intensive — uses thousands of GPUs/TPUs for weeks or months.
- **Output**: A base FM with broad capabilities but no task-specific optimization.
- Most organizations **skip this stage** and start with an existing pre-trained FM.

---

### Stage 4: Fine-Tuning
- Further train the pre-trained FM on **task-specific, labeled data** to specialize behavior.
- **Types**: Instruction tuning, domain adaptation, RLHF, PEFT/LoRA (see Task 3.3 for details).
- **Output**: A model customized for a specific task, domain, or style.
- Much cheaper than pre-training but still requires labeled data and compute.
- **AWS**: Amazon Bedrock fine-tuning, Amazon SageMaker.

---

### Stage 5: Evaluation
- Assess model quality using automated metrics and human evaluation before deployment.
- **Automated**: ROUGE, BLEU, BERTScore, benchmark datasets.
- **Human**: Preference ranking, Likert scale ratings.
- **AWS**: Amazon Bedrock Model Evaluation.
- **Key questions to answer**:
  - Does the model meet accuracy thresholds?
  - Is it safe, unbiased, and aligned with guidelines?
  - Does it outperform the baseline or previous model version?
  - Does it meet latency and cost SLAs?

---

### Stage 6: Deployment
- Make the model available for inference in production.
- **Deployment options**:
  - **Real-time endpoints**: Low-latency, always-on serving for interactive apps.
  - **Batch inference**: Process large volumes of data asynchronously.
  - **Serverless inference**: Auto-scales to zero; cost-efficient for sporadic workloads.
- **AWS services**:
  - **Amazon Bedrock**: Managed FM deployment with built-in scaling and guardrails.
  - **Amazon SageMaker**: Deploy custom models to managed endpoints.
- **Deployment considerations**:
  - Autoscaling to handle traffic spikes.
  - Guardrails and content filtering.
  - Monitoring for latency, errors, and cost.
  - A/B testing for model version comparisons.

---

### Stage 7: Feedback and Monitoring (Continuous Improvement)
- Collect real-world feedback to identify failures and improve the model over time.
- **Feedback sources**:
  - **Explicit**: User thumbs up/down ratings, correction submissions.
  - **Implicit**: Engagement metrics, re-prompting behavior, task abandonment.
  - **Automated**: Drift detection, output quality monitoring.
- **Monitoring dimensions**:
  - Output quality drift (is the model degrading over time?)
  - Data drift (is the production input distribution shifting from training?)
  - Safety violations and harmful outputs
  - Latency and throughput performance
- **Feedback loop**:
  - Collected feedback feeds back into data selection and fine-tuning for the next model version.
  - RLHF is a formalized version of using human feedback to improve the model.
- **AWS**: Amazon CloudWatch, Amazon Bedrock model monitoring, SageMaker Model Monitor.

---

### FM Lifecycle Stage Summary

| Stage | Key Activity | AWS Service |
|---|---|---|
| **Data Selection** | Collect, clean, and curate training data | Amazon S3, AWS Glue, SageMaker Ground Truth |
| **Model Selection** | Choose architecture, size, and starting point | Amazon Bedrock (FM catalog), SageMaker |
| **Pre-Training** | Train on massive unlabeled dataset | SageMaker (custom), AWS Trainium |
| **Fine-Tuning** | Specialize on labeled task-specific data | Amazon Bedrock, SageMaker |
| **Evaluation** | Assess quality, safety, and performance | Amazon Bedrock Model Evaluation, SageMaker |
| **Deployment** | Serve model for real-time or batch inference | Amazon Bedrock, SageMaker Endpoints |
| **Feedback** | Monitor outputs and collect improvement signals | CloudWatch, Bedrock Monitoring, SageMaker Model Monitor |
